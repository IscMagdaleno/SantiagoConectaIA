# Metodología Engrama: Capa de Cliente (Blazor WebAssembly PWA)

La capa de **Cliente** en la metodología Engrama utiliza **.NET Blazor WebAssembly** en su versión más reciente como tecnología principal para desarrollar una interfaz gráfica moderna, responsiva y funcional, configurada como una **Progressive Web App (PWA)**. Esta capa permite a los usuarios interactuar con la aplicación a través de componentes visuales, aprovechando la biblioteca **MudBlazor** para simplificar el desarrollo de componentes funcionales y estilizados. Este documento detalla la estructura, convenciones y ejemplos para el cliente, asegurando homogeneidad con las capas de Servicio, Dominio e Infraestructura.

## Tecnologías y Herramientas

- **.NET Blazor WebAssembly**: Framework utilizado para crear aplicaciones web interactivas, optimizadas para ejecución en el navegador.
- **MudBlazor**: Biblioteca de componentes que proporciona elementos de interfaz predefinidos, como tablas, formularios y barras de notificación, para acelerar el desarrollo.
- **EngramaCoreStandard**: Paquete que incluye servicios y clases estándar de Engrama, como `IHttpService`, `MapperHelper`, y `IValidaServicioService`, para estandarizar la interacción con la API y la lógica del cliente.

## Estructura del Cliente

La estructura de carpetas en el proyecto cliente está diseñada para mantener la modularidad y alinearse con los módulos definidos en las capas inferiores. Las carpetas principales son:

1. **wwwroot**:
   - Contiene archivos estáticos como CSS, JavaScript e imágenes necesarios para el diseño y funcionamiento de la aplicación.
   - Ejemplo: Archivos de estilos personalizados o scripts de terceros.

2. **Modules**:
   - Organiza la aplicación en submódulos, cada uno representando una sección funcional (por ejemplo, `Usuarios`, `Productos`).
   - Cada submódulo contiene:
     - **Pages**: Archivos `.razor` que representan páginas completas, con su contraparte `.razor.cs` para la lógica del backend.
     - **Components**: Archivos `.razor` que contienen componentes reutilizables administrados por las páginas.
     - **Main Class**: Una clase con prefijo `Main` (por ejemplo, `MainUsuarios`) que centraliza la lógica del módulo, incluyendo llamadas a la API y manejo de datos.

3. **Utilities**:
   - Contiene clases y herramientas reutilizables que soportan la funcionalidad de las páginas y componentes, como la clase `Main` de cada módulo.

### Convenciones para Páginas

- **Nomenclatura**: Cada página se nombra con el prefijo `Page` seguido de la funcionalidad (por ejemplo, `PageCrudUsuarios.razor` y `PageCrudUsuarios.razor.cs`).
- **Herencia**: Las páginas heredan de `EngramaPage` tanto en el archivo `.razor` (con `@inherits EngramaPage`) como en el `.razor.cs` (con `: EngramaPage`). Esto proporciona acceso a métodos y servicios estándar de Engrama.
- **Propósito**: Las páginas organizan y administran componentes relacionados (por ejemplo, `TableUsuarios.razor`, `FormUsuarios.razor`) para cumplir con las necesidades del módulo.

### Convenciones para Componentes

- **Nomenclatura**: Los componentes se nombran según su funcionalidad (por ejemplo, `TableUsuarios.razor`, `FormUsuarios.razor`) y se almacenan en una subcarpeta `Components` dentro del módulo.
- **Herencia**: Los componentes heredan de `EngramaComponent` (con `@inherits EngramaComponent`), proporcionando acceso a funcionalidades comunes como notificaciones y estados de carga.
- **Propósito**: Los componentes son elementos reutilizables que implementan partes específicas de la interfaz, gestionados por las páginas.

## Clase Main: Centralización de Lógica

Cada módulo incluye una clase con prefijo `Main` (por ejemplo, `MainTest`) en la carpeta `Utilities`. Esta clase centraliza la lógica del módulo, incluyendo:

- **Llamadas a la API**: Gestiona las solicitudes a los endpoints de la capa de Servicio.
- **Almacenamiento de Datos**: Mantiene los objetos utilizados por las páginas y componentes, facilitando la transferencia de datos entre ellos.
- **Dependencias**: Utiliza servicios de Engrama como `IHttpService`, `MapperHelper`, y `IValidaServicioService`.

### Ejemplo de Clase Main

```csharp
public class MainTest
{
    private readonly string url = "api/Test";
    private readonly IHttpService httpService;
    private readonly MapperHelper mapperHelper;
    private readonly IValidaServicioService validaServicioService;

    public IList<TestTable> LstTestTable { get; set; } = new List<TestTable>();

    /// <summary>
    /// Inicializa la clase con las dependencias necesarias para el módulo Test.
    /// </summary>
    public MainTest(IHttpService httpService, MapperHelper mapperHelper, IValidaServicioService validaServicioService)
    {
        this.httpService = httpService;
        this.mapperHelper = mapperHelper;
        this.validaServicioService = validaServicioService;
    }
}
```

#### Explicación
- **URL**: Define la ruta base del controlador correspondiente en la capa de Servicio (por ejemplo, `api/Test`).
- **Dependencias**:
  - `IHttpService`: Servicio de EngramaCoreStandard para realizar solicitudes HTTP a la API.
  - `MapperHelper`: Herramienta para mapear objetos entre modelos (por ejemplo, de `PostModel` a entidades de la API).
  - `IValidaServicioService`: Servicio para validar respuestas de la API.
- **Datos**: Propiedades como `LstTestTable` almacenan los datos del módulo, accesibles por páginas y componentes.

### Ejemplo de Método en la Clase Main

El siguiente método consulta la API para obtener una lista de datos:

```csharp
public async Task<SeverityMessage> PostTestTable()
{
    var apiUrl = $"{url}/PostTestTable";
    var model = new PostTestTable();

    var response = await httpService.Post<PostTestTable, Response<List<TestTable>>>(apiUrl, model);
    var validation = validaServicioService.ValidacionServicio(response, 
        onSuccess: data => LstTestTable = data.ToList());

    return validation;
}
```

#### Explicación
- **URL Completa**: Combina la ruta base (`url`) con el endpoint específico (`PostTestTable`).
- **Modelo**: Crea un `PostTestTable` con los parámetros necesarios para la solicitud.
- **Solicitud HTTP**: Usa `IHttpService.Post` para enviar la solicitud, recibiendo un `HttpResponseWrapper<Response<List<TestTable>>>`.
- **Validación**: `IValidaServicioService.ValidacionServicio` procesa la respuesta, ejecutando la acción `onSuccess` para actualizar `LstTestTable` si la solicitud es exitosa.
- **Respuesta**: Devuelve un `SeverityMessage` para notificar el resultado al componente o página.

## Clases de Respuesta

### SeverityMessage
La clase `SeverityMessage` estandariza las respuestas del cliente, heredando de `GenericResponse`:

```csharp
public class SeverityMessage : GenericResponse
{
    public string Position { get; }
    public SeverityTag Severity { get; }

    public SeverityMessage(bool bContinueProcess, string vchMessage, SeverityTag severity)
        : base(bContinueProcess, vchMessage)
    {
        Severity = severity;
    }

    public SeverityMessage(bool bResult, string vchMessage)
        : base(bResult, vchMessage)
    {
    }
}
```

### GenericResponse
La clase base para respuestas:

```csharp
public class GenericResponse
{
    public bool bResult { get; set; }
    public string vchMessage { get; set; } = "Sin Resultado";

    public GenericResponse() { }

    public GenericResponse(bool bResult, string vchMessage)
    {
        this.bResult = bResult;
        this.vchMessage = vchMessage;
    }
}
```

### HttpResponseWrapper
La clase que encapsula las respuestas HTTP:

```csharp
public class HttpResponseWrapper<T>
{
    public bool Success { get; set; }
    public T Response { get; set; }
    public HttpResponseMessage HttpResponseMessage { get; set; }

    public HttpResponseWrapper(T response, bool success, HttpResponseMessage httpResponseMessage)
    {
        Success = success;
        Response = response;
        HttpResponseMessage = httpResponseMessage;
    }

    public async Task<string> GetBody()
    {
        return await HttpResponseMessage.Content.ReadAsStringAsync();
    }
}
```

## Integración con Páginas y Componentes

### Ejemplo de Página (.razor.cs)

```csharp
public partial class PageTestModule : EngramaPage
{
    public MainTest Data { get; set; }

    protected override void OnInitialized()
    {
        Data = new MainTest(httpService, mapperHelper, validaServicioService);
    }
}
```

### Ejemplo de Página (.razor)

```html
@using TemplatePWA.PWA.Areas.TestModule.Components
@inherits EngramaPage
@page "/PageTestModule"

<MudCard>
    <MudCardHeader>
        <CardHeaderContent>
            <MudText Typo="Typo.h6">Test Page</MudText>
        </CardHeaderContent>
    </MudCardHeader>
    <MudCardContent>
        <TableTestTable Data="@Data" />
    </MudCardContent>
</MudCard>
```

#### Explicación
- **Herencia**: La página hereda de `EngramaPage`, proporcionando acceso a servicios como `httpService` y métodos como `ShowSnake`.
- **Inicialización**: Crea una instancia de `MainTest` en `OnInitialized`, que se pasa como parámetro (`Data`) al componente hijo.
- **Componente**: El componente `TableTestTable` recibe la instancia de `MainTest` para acceder a sus datos y métodos.

### Ejemplo de Componente (.razor.cs)

```csharp
public partial class TableTestTable : EngramaComponent
{
    [Parameter] public MainTest Data { get; set; }

    protected override async Task OnInitializedAsync()
    {
        Loading.Show();
        var result = await Data.PostTestTable();
        ShowSnake(result);
        Loading.Hide();
    }
}
```

### Ejemplo de Componente (.razor)

```html
@using TemplatePWA.Share.Objects
@inherits EngramaComponent

<MudTable Items="@Data.LstTestTable"
          T="TestTable"
          Dense="true"
          Hover="true"
          RowsPerPage="15"

        <MudTd DataLabel="dtRegistered">@context.dtRegistered</MudTd>

```

#### Explicación
- **Herencia**: El componente hereda de `EngramaComponent`, proporcionando acceso a servicios como `Loading` y `Snackbar`.
- **Parámetro**: Recibe la instancia de `MainTest` como `Data`, accediendo a `LstTestTable` para mostrar datos.
- **Inicialización**: Llama a `PostTestTable` en `OnInitializedAsync` y muestra un mensaje con `ShowSnake` basado en `SeverityMessage`.

## Clases Base: EngramaPage y EngramaComponent

### EngramaPage

```csharp
public class EngramaPage : ComponentBase
{
    [Inject] protected IValidaServicioService validaServicioService { get; set; }
    [Inject] protected IHttpService httpService { get; set; }
    [Inject] protected MapperHelper mapperHelper { get; set; }
    [Inject] protected LoadingState Loading { get; set; }
    [Inject] protected ISnackbar Snackbar { get; set; }

    /// <summary>
    /// Muestra una notificación en la interfaz usando MudBlazor.
    /// </summary>
    public void ShowSnake(SeverityMessage severityMessage)
    {
        if (!string.IsNullOrEmpty(severityMessage.vchMessage))
        {
            Snackbar.Clear();
            Snackbar.Configuration.PositionClass = Defaults.Classes.Position.TopCenter;
            Snackbar.Add(severityMessage.vchMessage, MudBlazorConverter.ConvertSeverity(severityMessage.Severity));
        }
    }
}
```

### EngramaComponent

```csharp
public class EngramaComponent : ComponentBase
{
    [Inject] protected LoadingState Loading { get; set; }
    [Inject] protected ISnackbar Snackbar { get; set; }

    /// <summary>
    /// Muestra una notificación en la interfaz usando MudBlazor.
    /// </summary>
    public void ShowSnake(SeverityMessage severityMessage)
    {
        if (!string.IsNullOrEmpty(severityMessage.vchMessage))
        {
            Snackbar.Clear();
            Snackbar.Configuration.PositionClass = Defaults.Classes.Position.TopCenter;
            Snackbar.Add(severityMessage.vchMessage, MudBlazorConverter.ConvertSeverity(severityMessage.Severity));
        }
    }
}
```

#### Explicación
- **EngramaPage**: Proporciona servicios como `httpService`, `mapperHelper`, y `validaServicioService` para páginas, además del método `ShowSnake` para notificaciones.
- **EngramaComponent**: Proporciona servicios como `Loading` y `Snackbar` para componentes, con el mismo método `ShowSnake`.

## Recomendaciones

- **Modularidad**: Mantenga los módulos pequeños y enfocados en una sola funcionalidad para facilitar el mantenimiento.
- **Reutilización**: Aproveche la clase `Main` para centralizar la lógica y evitar duplicación en páginas y componentes.
- **Validaciones**: Agregue validaciones en los métodos de la clase `Main` (por ejemplo, verificar que el `PostModel` no sea nulo antes de enviar la solicitud).
- **Pruebas**: Desarrolle pruebas unitarias para los métodos de la clase `Main` y pruebas de integración para las páginas y componentes.
- **Estilo**: Personalice los estilos en `wwwroot` para alinear la interfaz con la identidad visual del proyecto, complementando los componentes de MudBlazor.

## Conclusión

La capa de Cliente en la metodología Engrama, basada en Blazor WebAssembly PWA y MudBlazor, proporciona una interfaz gráfica robusta y modular. La estructura de carpetas (`wwwroot`, `Modules`, `Utilities`), la herencia de `EngramaPage` y `EngramaComponent`, y la centralización de lógica en la clase `Main` aseguran homogeneidad, reutilización y facilidad de mantenimiento. Al estandarizar las respuestas con `SeverityMessage` y centralizar las llamadas a la API, el cliente se integra eficientemente con las capas inferiores, ofreciendo una experiencia de usuario consistente y escalable.
